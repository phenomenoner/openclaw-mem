# Part 1｜記憶太用力：LLM 個人化為什麼開始讓人煩

Andrej Karpathy 最近在 X 上講了一個很多人都有感、但很少被說準的問題：現在很多 LLM 的 personalization memory 不是太弱，而是太煩。

他的原話大意是：兩個月前你只是隨口問了一次某個題目，模型之後卻一直把它當成你的核心興趣，時不時就要扯回來，像是在很努力地證明「我記得你」。他用了一個很準的詞：**trying too hard**。

這句話值得停一下。因為它點到的，不是大家最常講的那個問題。

大家平常談 LLM 記憶，焦點多半放在「忘得太快」：
- context window 不夠長
- 多輪對話後前面資訊掉光
- 一跨 session 就失憶
- agent 做久了沒有 continuity

這些都是真的。但現在另一個問題開始變得一樣重要：**它也會記得太用力。**

不是沒有記住，而是把不該長期占空間的東西留下來；不是沒有調用記憶，而是過度調用；不是沒有個人化，而是把個人化做成一種持續干擾。

換句話說，今天很多令人不舒服的 memory 體驗，不是 forgetting，而是 **over-remembering**、**over-applying**、**over-mentioning**。

## 這不是模型突然有了人格，而是產品層開始敢長期留住上下文
如果把問題講成「LLM 自己不會管記憶」，這句話在工程上大致沒錯。大部分所謂的 personalized memory，其實不是 base model 本身長出了一套可靠的長期個人記憶，而是產品層做了這些事：

- 把某些互動寫進 memory store
- 做分類、索引、embedding、rank
- 在下一輪或下一次 session 檢索出來
- 再把它塞回 prompt 或回應策略裡

所以 Karpathy 指到的問題，與其說是某個模型突然太油，不如說是這整條 memory orchestration 開始變得太積極。

近一年這個問題特別明顯，原因也很直接：
- persistent memory 越來越常見
- agent 越來越常跨 session 工作
- 產品越來越想做出 continuity 與「懂你」的感覺
- 大 context 讓人誤以為只要有更多舊內容，回答就會更好

於是很多系統都開始踩到同一顆雷：**把能記住，誤當成該記住；把能提起，誤當成該提起。**

## 真正的問題：不是一條 bug，而是整個 memory stack 都在偏向過度反應
如果只說「這是因為沒有 decay 機制」，方向沒有錯，但不夠。因為很多糟糕體驗不是單靠時間衰減就能救。

比較完整的診斷，我會拆成四層。

### 1. 寫入太寬
很多系統太容易把弱訊號寫成 durable memory。

例如：
- 使用者隨口問一次京都自由行
- 使用者某天臨時研究一個冷門框架
- 使用者對某個主題短暫爆量提問，但其實是因為當週的工作，不是長期偏好

這些都很容易被存進去，然後從「一次對話中的偶發訊號」升格成「關於這個人的長期資訊」。

一旦寫進去，後面的每一層都要替這個決定付代價。

### 2. 分類太粗
很多 memory 系統把所有東西都當成同一種東西：
- 穩定偏好
- 長期身份資訊
- 近期任務狀態
- 一次性的對話痕跡
- 文件知識
- 原始工具輸出

這些如果混在一起，後果很簡單：
- retention policy 沒法細分
- retrieval competition 會失真
- 一次性痕跡跟長期偏好會搶同一個 recall budget

最後你就會得到一種很怪的感覺：
明明模型記的是「真的發生過的事」，但它拿來用的方式完全不對。

### 3. 檢索太鬆
這是很多系統最常被忽略的一層。

它未必真的很懂你，只是只要語義上有點像，就把舊記憶拖回來。

如果 retrieval policy 太寬：
- 兩個月前那次日本旅遊問題
- 今天聊生活節奏、咖啡、休假、甚至任何地點
都可能被當成相近主題而召回。

於是系統不是「理解了這是你的長期興趣」，而是「在很寬鬆的相似度條件下，不斷把同一條舊訊號重新撈上來」。

### 4. 表達太前台
這其實是最讓人煩的一層。

就算記憶是正確的、召回也是合理的，系統仍然需要再過一道判斷：

> **現在把這件事說出口，是真的有幫助，還是只是在表演我記得你？**

很多產品沒有這層。
只要 recall 了，就默默讓它進 prompt，甚至直接在回答裡點名。

於是你看到的不是「這系統很貼心」，而是：
- 它每次都要 cue 那件舊事
- 它過度連結你的現在和你的過去
- 它把 continuity 做成一種存在感過強的角色扮演

Karpathy 那句 **trying too hard**，打中的就是這一層。

## 所以問題不是「記憶對不對」，而是「提起的比例對不對」
這也是為什麼我很同意一種進一步的說法：

> 記憶內容是對的，不代表提起時機是對的。

這句話很關鍵。
因為很多人直覺會把問題分成兩種：
- 記對
- 記錯

但真實世界還有第三種，而且可能更常見：
- **記對，但用錯。**

比如：
- 這件事的確聊過
- 的確寫進 memory 了
- 的確跟現在題目沾上一點邊

但這一刻提起它，仍然是錯的 UX。

這才是 personalization 系統最麻煩的地方：
它不只是 retrieval correctness 的問題，而是 **context proportionality** 的問題。

## 為什麼 decay 不是完整答案
很多人第一反應是：那就加 decay。

這方向是對的，但它不是主菜，比較像垃圾回收。

Decay 能解的事：
- 降低舊訊號長期霸榜
- 讓一次性痕跡自然淡出
- 限制資料庫膨脹

Decay 解不了的事：
- 一開始就寫錯類型
- 檢索規則太寬
- 召回後沒有 suppression / mention gate
- 低信任內容也能重新進 prompt

所以如果把整件事講成「人類記憶會淡化，但 LLM 沒有衰減」，那其實還是講得太乾淨了。現實更像是：

- 有些系統寫太多
- 有些系統分太粗
- 有些系統撈太鬆
- 幾乎所有系統都太少談「該不該說出口」

## 這也是為什麼「可控」比「自動」更重要
黑盒 auto-memory 最讓人不安的，不只是它會記什麼，而是你不知道：
- 它為什麼記這條
- 它為什麼現在提這條
- 它什麼時候才會不再提

所以相較之下，像可編輯文字記憶、清楚的 daily notes、可審計的 receipts，雖然不一定自動化最強，卻通常更容易讓人建立信任。

原因不是它比較聰明，而是它比較誠實。

你可以看到它記了什麼，改它、刪它、懷疑它。
這件事本身，就是治理的一部分。

## 這個問題真正逼我們面對的是：好的 memory 系統要會沉默
如果要把這篇收成一句話，我會這樣講：

> 現在 personalization memory 最常見的失敗，不是忘記你，而是太努力地表演自己記得你。

一個好的記憶系統，不是記得越多越厲害。
它真正厲害的地方是：
- 知道哪些東西不該被寫成長期訊號
- 知道哪些東西雖然能召回，但不該占用這一輪 context
- 知道哪些東西就算 relevant，也不需要被直接說出口

換句話說，**好的 memory 系統不是更會想起，而是更會克制。**

下一篇我們就來拆：大家已知的解法到底有哪些，它們各自解的是哪一層，又是哪裡還不夠。
