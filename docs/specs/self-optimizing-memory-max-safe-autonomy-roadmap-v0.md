# Spec — self-optimizing memory max-safe autonomy roadmap v0

## Status
- Stage: planning reference after bounded autonomy Phase 5
- Scope: move from **bounded autonomous low-risk evolution** toward a system that is closer to the safe autonomy ceiling without collapsing governance
- Execution mode: `櫻花刀舞` blade map for post-Phase-5 serial push

## Verdict
Phase 5 is a real milestone, but it is **not** the ceiling.
The next line is not “more courage”.
It is building the missing proof loops so autonomy can widen **only when the system can measure, challenge, pause, and explain itself**.

## Whole-picture promise
A near-upper-bound safely autonomous evolving memory system should be able to:
- improve itself across multiple mutation families
- verify whether changes helped over time, not only at write time
- pause itself on regression before human review catches it manually
- widen or narrow its autonomy envelope with explicit receipts
- preserve the scout / governor / writer split even when much more runs unattended

Fake progress would be:
- widening mutation authority without follow-up proof
- adding more metrics that do not actually gate behavior
- calling a canary controller “max-safe” while it still depends on operator-crafted receipts for core truth

## Current baseline (already shipped)
The current system already has:
- bounded low-risk apply classes
- evidence-weighted classifier foundation
- family caps
- persisted controller state machine
- watchdog-triggered self-pause
- promotion gates into `auto_low_risk`
- governor review retained for medium-risk
- proposal-only posture retained for high-risk

This is the right base.
It is still **bounded low-risk autonomy**, not a near-ceiling autonomy system.

## What is still missing

### Gap 1 — follow-up effect loop is still shallow
Current effect receipts are mostly baseline / immediate-run oriented.
Missing:
- delayed re-measurement
- multi-horizon comparison
- family-specific quality deltas
- action-level success/failure memory

### Gap 2 — rollback replay is not yet a native always-on verifier lane
Current posture can consume rollback replay receipts.
Missing:
- built-in rollback replay job
- deterministic fixture/live-sample replay cadence
- controller-native gating from replay output

### Gap 3 — promotion truth is still partly operator-fed
Current promotion works, but still expects external gate receipts for key metrics.
Missing:
- first-party gate artifact production
- native measurement commands for promotion metrics
- multi-cycle soak evidence generated by the system itself

### Gap 4 — mutation-family coverage is still narrow
Current autonomy mostly covers observation hygiene.
Missing:
- broader but still bounded mutation classes
- stronger family-specific contracts
- per-family verifier lanes before promotion

### Gap 5 — there is no challenger / adversarial lane yet
Current controller can pause on obvious regressions.
Missing:
- shadow challenger policies
- compare-and-veto behavior between primary and challenger policies
- regression attribution rather than only regression detection

## Roadmap

### Phase 6 — native effect follow-up loop
Goal: make autonomy judge outcomes over time, not only emit baseline receipts.

Ship:
- first-party follow-up measurement command for 24h / 72h windows
- effect receipt enrichment with family-specific `quality_delta`
- per-action success history ledger
- controller consumption of follow-up effect summaries, not just baseline receipts

Exit criteria:
- follow-up receipt generation is automatic for applied runs
- `improved | neutral | regressed | insufficient_data` is based on actual re-measurement
- effect receipt missing rate stays zero across multiple cycles

Recommended first slice:
- add a read-only `optimize effect-followup` surface that ingests prior apply receipts and emits delayed effect receipts without changing mutation authority

### Phase 7 — native verifier lane
Goal: internalize the proof loop so the system no longer depends on hand-fed rollback/promotion artifacts for core gates.

Ship:
- built-in rollback replay command + receipt
- deterministic sample-db verifier lane
- periodic verifier bundle artifact under runner root
- controller gate wiring to native verifier results

Exit criteria:
- rollback replay pass/fail is generated by the system itself
- verifier receipts are deterministic and reviewable
- controller pause logic can run without operator-crafted rollback receipts

Recommended first slice:
- `optimize verifier-bundle --json` emitting rollback replay pass, effect-receipt completeness, and cap-integrity checks

### Phase 8 — policy challenger lane
Goal: let the system test competing policy ideas safely before widening authority.

Ship:
- shadow challenger classifier / policy mode
- compare primary vs challenger on the same candidate stream
- veto / quarantine when challenger shows better safety or regression detection
- receipts for disagreement clusters

Exit criteria:
- challenger runs remain strictly read-only at first
- disagreement packets are auditable and bounded
- controller can require challenger agreement before promoting a new family/rule

Recommended first slice:
- read-only challenger packet comparing current classifier to an alternate scoring policy on the same evolution-review output

### Phase 9 — bounded family expansion
Goal: widen autonomy beyond hygiene without abandoning contracts.

Candidate families:
- bounded importance upshift with stronger reuse evidence
- score-label repair as pure consistency action
- bounded archive candidate state metadata
- bounded duplicate-cluster proposal metadata

Ship requirements for each family:
- exact whitelist
- family-specific classifier rules
- family-specific rollback replay tests
- family-specific effect metrics
- promotion held per family, not globally hand-waved

Exit criteria:
- each new family has its own proof pack
- family rollout is reversible independently
- controller supports family-level enable/disable state

Recommended first slice:
- promote score-label alignment into a separately-gated family with zero semantic broadening

### Phase 10 — multi-horizon autonomy controller
Goal: make the controller reason over short, medium, and soak windows instead of one recent window.

Ship:
- controller state extended with short-horizon, medium-horizon, and soak status
- promotion requires multi-cycle green state, not one receipt
- demotion path from `auto_low_risk` back to `canary_apply` or `dry_run`
- incident memory / pause reasons with expiry and acknowledgement semantics

Exit criteria:
- controller supports promotion, demotion, and quarantine explicitly
- repeated small regressions accumulate into action, not silence
- pause/unpause is no longer a blunt binary toggle only

Recommended first slice:
- add controller substate for `soak_green_cycles` and `regression_strikes`

### Phase 11 — near-ceiling safe autonomy posture
Goal: approach the safe ceiling without claiming unbounded autonomy.

This posture means:
- the system generates its own effect, verifier, and promotion evidence
- multiple bounded mutation families can run unattended
- policy changes are challenged before activation
- regression demotion is automatic
- human review shifts to medium/high-risk and governance deltas, not daily babysitting

This posture still does **not** mean:
- unbounded self-rewrite
- prompt/config/model-routing self-mutation
- broad cross-scope semantic surgery
- deletion/merge/split autonomy without a stronger theorem of safety

Exit criteria:
- multiple cycles of green follow-up + verifier + challenger evidence
- family-level autonomy can be widened/narrowed without code surgery
- incident/pause/demotion receipts are complete enough for postmortem

## Cross-cutting contract rules

### 1. Family-level, not global hand-waving
Every mutation family must carry its own:
- whitelist
- risk ladder
- verifier
- effect metric
- cap set
- promotion state

### 2. Promotion must be multi-cycle
No single green receipt should be enough for meaningful widening.
Promotion should require soak evidence across multiple cycles.

### 3. Demotion must be first-class
A safe autonomy system needs explicit downgrade paths:
- `auto_low_risk -> canary_apply`
- `canary_apply -> dry_run`
- any active mode -> `paused_regression`

### 4. Native evidence beats hand-fed evidence
External/operator-fed receipts are acceptable during ramp-up.
They should be retired as first-party verifier and effect lanes become real.

### 5. Governance split must survive success
Even near the autonomy ceiling:
- scout still scouts
- governor still judges wider-risk moves
- writer still writes only inside an explicit contract

## Suggested execution order
1. **Phase 6** native effect follow-up loop
2. **Phase 7** native verifier lane
3. **Phase 8** challenger lane
4. **Phase 9** bounded family expansion
5. **Phase 10** multi-horizon controller
6. **Phase 11** near-ceiling posture review

Recommendation:
Do **not** widen mutation families before Phase 6 and Phase 7 are real.
That would be fake acceleration.

## Tradeoffs / open risks
- More proof lanes mean more moving parts and more artifacts to keep coherent.
- If we widen mutation families before native verifier/effect loops exist, the system will look smarter while becoming less safe.
- The safe ceiling is not unbounded power. It is the point where unattended behavior remains explainable, challengeable, and reversible.
