# Self Curator apply-capable v1 — checkpointed mutation contract

Status: **SHIPPED / UPDATED FOR UNATTENDED BODY MUTATION**

v0 remains a useful review-only scout/proposal surface. v1 is apply-capable: it may directly mutate whitelisted files only through a checkpointed protocol.

## Goal

Build a Hermes-curator-like Self Curator that may directly change files or settings **inside an explicit checkpointed apply protocol**.

The contract is not "never mutate". The contract is:

> mutate only through a plan → checkpoint → apply → verify → rollback-capable receipt loop.

## Whole-picture promise

Self-improvement should be able to metabolize old skills, memory policy, and eventually safe settings without forcing CK to hand-edit every small hygiene change, while still making every change inspectable and reversible.

## Posture layers

```text
review-only scout
  -> apply plan
  -> rollback checkpoint
  -> apply whitelisted mutation
  -> verify/readback
  -> human-readable report
  -> WAL + receipt
  -> rollback command remains available
```

## v1 command surface

```bash
openclaw-mem self-curator skill-review ...
openclaw-mem self-curator plan --mutations-file mutations.json --out plan.json --workspace-root . --json
openclaw-mem self-curator apply --plan plan.json --workspace-root . --checkpoint-root .state/self-curator/checkpoints --receipt-root .state/self-curator/apply-runs --json
openclaw-mem self-curator verify --receipt .state/self-curator/apply-runs/<run>/apply-receipt.json --json
openclaw-mem self-curator rollback --receipt .state/self-curator/apply-runs/<run>/apply-receipt.json --json
openclaw-mem self-curator controller --mode unattended_apply ... --json
```

## Scheduled unattended controller policy

The cron/controller path is intentionally narrower than the generic apply engine.

Allowed unattended mutation families today:

1. **Bounded skill body lifecycle section append**

   The controller may append exactly this section shape to `skills/<name>/SKILL.md`:

   ```md
   ## Curator lifecycle

   - Status: `promote_to_review|refresh|review`
   - Reason: `<reason_code>`
   - Rollback: use the apply receipt generated by `openclaw-mem self-curator`.
   ```

   Requirements:
   - target must be `*/SKILL.md` under workspace;
   - mutation action is `replace_text`;
   - patch must append the exact bounded section shape;
   - section must not already exist;
   - precondition must include target SHA-256;
   - repeated controller runs must no-op after the first mutation.

2. **Checkpointed archive for malformed/very-short skill files**

   The controller may archive a `SKILL.md` to:

   ```text
   skills/.archive/<skill-name>/SKILL.md
   ```

   This is allowed only for deterministic review reasons such as `very_short_skill_body` or `missing_frontmatter_name`. Rollback restores the original file bytes and removes the archive destination.

## Generic apply engine mutation classes

The lower-level apply engine supports explicit plans with these actions:

- `replace_text`
- `write_file`
- `move_file`
- `archive_file`
- `set_frontmatter_field`

Generic plans are not automatically scheduled unless admitted by the controller policy.

## Out of scope for unattended controller

- `SOUL.md`, `AGENTS.md`, `MEMORY.md`, runtime config, gateway config.
- Hard delete.
- LLM-authored freeform rewrites.
- External publish/post/PR without separate review.
- Broad skill prose rewrites not matching the exact lifecycle-section append.

## Apply plan schema

```json
{
  "kind": "openclaw.curator.apply-plan.v1",
  "mode": "apply_plan",
  "source_review": ".../review.json",
  "plan_id": "...",
  "ts": "...",
  "mutations": [
    {
      "mutation_id": "...",
      "target_ref": "skills/foo/SKILL.md",
      "action": "replace_text|write_file|move_file|archive_file|set_frontmatter_field",
      "risk_class": "skill_surface",
      "preconditions": {
        "sha256": "...",
        "must_contain": "..."
      },
      "patch": {
        "old_text": "...",
        "new_text": "..."
      },
      "rollback_strategy": "restore_checkpoint"
    }
  ],
  "requires_checkpoint": true
}
```

## Invariants

- Every apply must create a checkpoint before the first write.
- Every write must be listed in the plan.
- Every target must have a precondition hash or exact text precondition.
- Any precondition mismatch fails closed before writes.
- Exceptions and partial writes restore from checkpoint.
- Apply receipt must include before/after hashes and a diff artifact.
- Rollback must restore file bytes from checkpoint, not attempt inverse fuzzy edits.
- Move/archive destinations are hash-verified.
- Scheduled controller may mutate whitelisted skill surfaces; authority/canon surfaces remain later-gated.

## Verifier plan

1. Unit tests:
   - plan validation rejects path traversal and directory targets;
   - apply creates checkpoint before write;
   - precondition mismatch performs zero writes;
   - exceptions after partial write restore checkpoint;
   - rollback restores exact original SHA-256;
   - controller body lifecycle append mutates `SKILL.md` and rolls back;
   - controller repeats no-op after existing lifecycle section;
   - freeform lifecycle-looking append is rejected by policy;
   - archive controller behavior restores source and removes archive destination.
2. CLI/controller smoke:
   - fixture skill receives `## Curator lifecycle` section;
   - repeat run returns `NO_REPLY`;
   - rollback restores original hash.
3. Counterfactual smoke:
   - mutate target between plan and apply; apply must fail closed with `writes_performed=0`.
